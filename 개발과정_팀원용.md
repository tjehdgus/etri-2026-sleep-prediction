# 개발 과정 정리 (팀원용)

> 이 문서는 대회를 직접 돌리지 않은 팀원도 **"어떻게 진행했고, 무엇을 시도했고, 무엇이 왜 실패했는지"**를
> 이해할 수 있도록 쉬운 말로 정리한 것입니다. 전문용어는 처음 나올 때 풀어 씁니다.

---

## 0. 한 줄 요약

스마트폰·스마트워치 센서로 **그날 밤 수면 7개 지표(0/1)**를 맞히는 대회.
약 보름간 수십 개의 아이디어를 시도해, 점수(낮을수록 좋음)를 **0.61 → 0.5584**까지 끌어내렸고
순위는 **35등 → 약 26등**까지 올렸습니다. 그 이상은 우리가 **가지지 못한 데이터(전문 수면측정기 원본)** 때문에 막혔습니다.

---

## 1. 문제가 뭐였나

- **입력**: 한 사람의 하루치 휴대폰·워치 센서 기록 12종 (심박, 걸음, 화면 사용, 충전, 조도, 위치, 주변소리 등)
- **출력**: 그날 밤 수면에 대한 **7개의 예/아니오(0/1) 라벨**
  - **Q1·Q2·Q3** = 설문 (수면질·피로·스트레스). 기준이 **"그 사람 평소 평균과 비교"**해서 좋았나/나빴나.
  - **S1·S2·S3·S4** = 전문 수면측정기(Withings) 기준 충족 여부 (총수면시간·수면효율·잠드는 시간·중간에 깬 시간이 의학 가이드라인을 지켰나).
- **데이터 크기**: 학습용 450일치 / 평가용 250일치. 사람은 **10명뿐**(같은 사람들이 학습·평가에 다 등장, 날짜만 다름).
- **채점 방식**: **로그 로스(log loss)** — 단순 정답률이 아니라 **"확신의 정도가 얼마나 잘 맞았나"**를 봅니다.
  자신 있게 틀리면 크게 감점. 그래서 **확률을 잘 보정**하는 게 핵심. (숫자가 낮을수록 좋음)

### 처음부터 있었던 근본적 한계
- S1~S4는 **전문 수면측정기로 잰 값**인데, 그 측정기 데이터는 우리에게 **주어지지 않았습니다.**
  우리는 휴대폰·워치로 그걸 **간접 추정**할 수밖에 없었고, 이게 끝까지 따라다닌 천장(상한선)이었습니다.
- 사람이 10명뿐이고 데이터가 적어서(450행) **조금만 복잡하게 만들면 과적합**(아래 용어 설명)이 나기 쉬웠습니다.

---

## 2. 핵심 개념 (용어 풀이)

| 용어 | 쉬운 설명 |
|---|---|
| **과적합 (overfitting)** | 학습 데이터의 우연한 패턴까지 외워버려서, **새 데이터에선 오히려 틀리는** 현상. 데이터가 적을수록 잘 일어남. |
| **CV (교차검증)** | 학습 데이터 일부를 가려놓고 맞혀보며 "실전 점수"를 미리 추정하는 방법. |
| **OOF 점수** | CV로 추정한 우리 내부 점수. **이게 거짓말을 하면** 실제 리더보드(LB)에서 망함. |
| **LB (리더보드)** | 대회 서버에 제출했을 때 나오는 진짜 점수. |
| **블렌드/앙상블** | 여러 모델의 예측을 **섞는 것**. 보통 단일 모델보다 안정적이고 점수가 좋아짐. |
| **LB transfer** | 우리 내부 점수 개선이 **실제 리더보드에서도 개선되는지** 여부. 안 되면 "환상(과적합)". |

---

## 3. 전체 여정 (시간 순)

점수는 **낮을수록 좋습니다.** 큰 흐름만 짚으면:

```
0.61대  →  0.586  →  0.5994  →  0.5960  →  0.5596  →  0.5584
(시작)    (레시피)   (다양성)   (이웃)    (라우팅/순위)  (forward-CV)
```

### (1) 0.61 → 0.586 : "제대로 된 레시피" 찾기
- 처음엔 인터넷 공유 코드를 따라 했더니 0.61대에서 막힘.
- **상위권 팀 발표자료를 분석**해 핵심 레시피를 적용:
  - 피처(특징) 선택을 **상관계수가 아니라 "모델이 중요하다고 본 것" 기준**으로 (ExtraTrees라는 모델 사용)
  - 부스팅 모델(LightGBM·CatBoost)에 **조기 종료**를 걸어 과적합 방지
  - 8천 개나 되던 피처를 **상위 150개만** 추림 (나머지는 노이즈였음)
  - **타겟(7개 라벨)마다 따로** 최적 조합을 찾음
- → 0.586까지 내려감. "0.60이 한계"라던 이전 판단이 틀렸음을 확인.

### (2) 0.586 → 0.5994 : "다른 시각" 더하기 (다양성)
- 아이디어: **센서 시계열을 이미지로 바꿔서**, 사진 인식용 AI(비전 모델)에 넣어보자.
- 비전 모델 단독 점수는 나빴지만(0.66~0.69), **트리 모델과 "보는 방식이 달라서"** 섞으면 도움이 됐습니다.
- 교훈: **서로 다른 방식의 모델을 섞을수록 좋다.** 단, 너무 많이 섞으면(10개+) 다시 과적합.

### (3) 0.5994 → 0.5960 : "수면 재구성" + "이웃 날짜"
- **수면 재구성(SleepTree2)**: 밤사이 심박·움직임·충전 기록으로 **"언제 잠들고 깼는지"를 역추적**해
  총수면시간·수면효율 같은 걸 추정 → S1~S4를 직접 겨냥. 효과 있었음.
- **이웃 날짜(Neighbor)**: 같은 사람의 **날짜가 가까운 다른 날 라벨**을 참고. 수면 습관은 날짜끼리 비슷하다는 점을 이용.
  센서가 못 잡는 직교(orthogonal, 겹치지 않는) 신호라 도움이 됐습니다.

### (4) 0.5960 → 0.5596 : "라벨 정의에 맞춘 학습"
- **daystate(일상상태 분해)**: 학습+평가 700일 전체를 라벨 없이 분석해 **"그 사람의 하루 패턴 유형"**을 뽑아냄.
  → 오래 막혀 있던 Q1·Q3을 처음으로 개선.
- **pairwise 순위학습**: Q 라벨이 **"개인 평균 대비"**라는 점에 착안.
  같은 사람의 (좋았던 날, 나빴던 날) **쌍을 비교**하도록 학습 → 정의에 딱 맞아 Q1·Q2 개선.
- 이 구간에서 점수가 크게 떨어져 **수상권 근처**까지 갔습니다.

### (5) 0.5596 → 0.5584 : "시간 이동을 인지한 검증" (마지막 결정적 레버)
- **발견**: 평가 데이터(test)는 학습 데이터보다 **뒤쪽 날짜**에 몰려 있었습니다(시간 이동).
  그런데 우리는 날짜를 **무작위로 섞어서** 검증하고 있었음 → 이 시간 이동을 못 봄.
- **해결(forward-CV)**: **"앞 날짜로 학습 → 뒤 날짜로 검증"** 방식으로 바꿔서,
  실전(test)과 똑같은 상황에서 S1·S4의 모델 섞는 비율을 다시 골랐습니다.
- 이게 마지막 큰 개선이었고, **이 비율을 더 강하게 밀수록 점수가 계속 좋아졌습니다**(0.5588 → 0.5586 → 0.5585 → 0.5584).
- 최종 **0.5584, 약 26등**에서 대회 종료.

## 4. 무엇이 통했나 (검증된 진짜 레버)

이 4가지만이 **리더보드에서도 실제로 개선**된 것들입니다:

1. **소스 다양성 앙상블** — 트리·딥러닝·비전·수면재구성·이웃 등 **방식이 다른 9개 모델**을 섞음.
2. **이웃 날짜(Neighbor)** — 같은 사람의 가까운 날 라벨 참고.
3. **pairwise 순위학습 / daystate 라우팅** — Q 라벨의 "개인 평균 대비" 정의에 학습 방식을 맞춤.
4. **DLens (분산 감소)** — 딥러닝 모델을 **여러 번(8개의 다른 랜덤 시드로) 학습해 평균**.
   딥러닝은 운(랜덤성)에 흔들리는데, 여러 번 돌려 평균 내면 안정적이 됨. (작게, 하지만 확실히 효과)
5. **forward-CV (시간 이동 인지)** — 위 (5)번. **가장 큰 마지막 레버.**

## 5. 무엇이 실패했나 (그리고 왜)

대회 내내 **"좋아 보였는데 실제로는 안 통한 것"**이 정말 많았습니다.
공통 원인은 거의 항상 둘 중 하나였습니다:
**(a) 과적합** 또는 **(b) 우리에게 없는 데이터를 억지로 추정하려 함.**

| 시도한 것 | 결과 | 왜 실패했나 |
|---|---|---|
| **연속 최적화 블렌드** (점수 깎으려 비율을 정밀 최적화) | 내부 0.599인데 LB 0.62로 **폭망** | 우리 내부 점수에만 맞춰져서 과적합. → **블렌드는 단순하게**가 교훈. |
| **모델·소스를 10개+로 잔뜩 추가** | 내부는 좋아지는데 LB 악화 | 많이 섞을수록 과적합. **적당한 다양성이 최적.** |
| **전문 수면측정기 값 정밀 복원** (여러 버전 v1~v3) | 전부 미세 악화 또는 제자리 | 우리는 **그 측정기 데이터가 없어서** 추정 정확도에 천장이 있음. |
| **HRV(심박변이도) 같은 정밀 생리 피처** | 기여 거의 0 | 이미 비슷한 정보(심박 통계)가 들어가 있어 **새 정보가 없었음.** |
| **피험자별 개인 모델(TabPFN 등)** | 효과 0 | 한 사람당 40일치뿐이라 **너무 적어서** 개별 모델이 못 배움. |
| **외부 날씨·WiFi·위치 핑거프린트** | LB에서 다 죽음 | 격리 실험에선 좋아 보였지만 실전 전이 실패(또 과적합). |
| **여러 "큰 베팅" (라벨 구조 활용, 정의서 기반 보정 등)** | 엄밀 검증하면 기여 ≈ 0 | 파이프라인이 이미 그 정보를 **다 짜내고 있어서** 더 얻을 게 없었음. |
| **데이터 정렬 버그 수정** (밤 시간대를 하루 잘못 끌어옴) | S1에선 효과 확인, 전체 개선은 제한적 | 신호는 맞았으나 마진이 작았음. |

### 가장 비싸게 배운 교훈
> **"우리 내부 점수(OOF)가 좋아져도, 리더보드에서 좋아진다는 보장이 없다."**
>
> 특히 **기존 최고 기록에서 예측값이 많이 달라지는(maxdiff가 큰) 후보는 거의 항상 리더보드에서 죽었습니다.**
> 반대로 **분산 감소(DLens)**나 **시간 이동 인지(forward-CV)**처럼,
> "새 정보를 우기는 게 아니라 **이미 있는 걸 더 안정적으로/정확한 기준으로 다루는**" 개선만 실제로 전이됐습니다.

## 6. 왜 더 못 갔나 (정직한 천장)

- 상위권(수상권)은 0.54~0.55대. 우리 최종은 0.5584.
- 그 격차의 핵심은 **S1~S4(전문 수면측정기 기준)**. 우리는 그 측정기 **원본 데이터가 없어서**
  휴대폰·워치로 간접 추정할 수밖에 없었고, 추정 정확도에 구조적 상한이 있었습니다.
- Q2·Q3(주관적 피로·스트레스)도 센서로 예측하는 데 본질적 한계가 있었습니다(약 0.69 벽).
- 수십 가지 방법을 **정직하게 검증**한 결과, 우리가 가진 12종 센서로는 **0.5584가 진짜 천장**이라고 결론냈습니다.
  그 이상은 **운(평가 데이터가 어떻게 섞이느냐, private 셔플)**에 기대는 수밖에 없었습니다.

## 8. 한 문단 회고

이 프로젝트는 **"좋은 아이디어를 찾는 것"보다 "통하는 아이디어와 환상을 구별하는 것"**이 훨씬 어려웠습니다.
수십 개를 시도해 대부분 버렸고, 진짜 살아남은 건 다섯 손가락 안에 들었습니다.
가장 큰 마지막 도약(0.5596→0.5584)은 더 똑똑한 모델이 아니라 **"평가 데이터가 시간적으로 이동했다"는 사실을
검증 방식에 반영한 것**에서 나왔습니다. 데이터의 구조를 정직하게 본 것이 모델을 더 키운 것보다 효과적이었습니다.
